# 定位、现阶段范围与获客 Backlog（权威梳理）

> 2026-07-08 定。给团队和未来 Claude 会话的**唯一权威「我们是谁、现在做什么、还剩什么、谁的活」**。
> 上游研究：[buyer-intelligence-v3.md](buyer-intelligence-v3.md)（10 支柱深研）· [roadmap-ai-acquisition.md](roadmap-ai-acquisition.md)（进度）。PRD：`docs/*.docx`（v2.0 母本 / v3.0 评审稿）。

## 1. 定位（三层）

> **我们是一台「线索工厂」的后端。** 输入=客户卖什么；输出=**一批打好分、有需求证据、找到对的人、合规能存的海外线索**。

| 层 | 内容 |
|---|---|
| **是什么（身份）** | 出海增长的**情报后端大脑**：多源发现 → 实体解析 → 富集核实 → 信号/意图/需求证据 → 决策人+邮箱 → 六维评分 → 存储侧合规。 |
| **不做什么（边界）** | ❌ 前端/界面 ❌ 用户/登录/权限/鉴权（身份系统）❌ 真正发信触达执行 ❌ 一企一套定制。这些属**另一位开发者的 SaaS 平台 + 触达执行系统**。 |
| **怎么交付（契约）** | ✅ 只交 **code-first OpenAPI 契约 + 领域事件 + Scalar 门户 `/api/portal`**。对接前端由**另一位「做 API 的人」**完成——我们给契约、他接前端，**我们不做联调本身、不为前端造 mock、不等前端**。 |

**我们对北极星的贡献是「上游」**：平台北极星=QGO（可成单商机）+ 单位成本；我们产的是 **QGO 的原料**（合格线索+需求证据+可达联系人+合规结论）。QGO 对象本身/成交回写/归因大概率在平台侧（见 §6 开放决策）。

## 2. 现阶段范围：只做 v3.0 的「获客」块

| | 内容 |
|---|---|
| ✅ **算获客（现在做）** | 挖公司 → 核实是不是合适买家 → 找需求证据（谁在买/招聘/招标/认证）→ 找决策人+验邮箱 → 六维打分排队 → **存这些人时的合规（存储侧）**。产出=「合格、可联系、可存」的线索。 |
| ⏸️ **不算获客（后延）** | 触达执行（发邮件/消息）、内容营销、**发信合规可发送性矩阵（B 门）**、商机 QGO 出口层、归因、市场研究、Campaign/Engage/Pack/Expert。等获客做透再说。 |

## 3. 业务方向 = 客户自选（多租户，不硬编码）

**目标国家/行业不是我们固定的**——由每个使用系统的客户（workspace）经 **ICP 自选**（多租户 RLS）。所有获客能力必须**按 workspace 的 ICP 参数化**，**绝不硬编码行业/国家**：

- TED 招投标：需「ICP 行业 → 欧盟 CPV 代码」映射层
- SAM.gov / 海关 HS / 认证库：需「ICP 行业 → NAICS / HS / 品类」映射
- SearXNG / 发现查询：按 ICP 国家+行业+关键词生成

## 4. 团队分工与架构（已定 2026-07-08）

**架构：单体、一个库**（现阶段业务简单先不拆，复杂了再拆；以 C 现有架构为基础——B 评审过，设计无问题）。
数据流：**A（前端）─▶ B（接口层，同库）──方法调用──▶ C + Claude（服务层，同库）**

| 角色 | 做什么 |
|---|---|
| **C + Claude** | **服务层 / 核心业务逻辑**：获客引擎（发现/富集/评分/provider/Temporal 编排/数据模型）。产出**干净、可被直接调用的服务方法** + 领域事件。 |
| **B** | **接口层 + API 契约**（controller / OpenAPI / BFF），在**同一个库**里**通过方法调用** C 的服务；对接 A。**现有 controller + `packages/contracts` 不删，B 复用/维护**。 |
| **A** | 前端**整个 SaaS 平台**（含身份/登录/鉴权）。A 经 B 连、**不直连我们**；A 的参数由 B 翻译成我们服务的参数。 |
| **C（用户，非技术）** | 给业务方向、拍板开放决策（§6）、传话 B/团队、审中文产出。不碰代码/git。 |

**深层编排归 C**：任务调度/爬虫/异步/重试用 **Temporal**（C 已建）；B 的接口层「启动任务 + 看进度」，**不重建引擎**。

**并行开发（B 已拉库、本地在做、需 push 权限）**：BC 同库并行；文件分工大致 **C = 服务/领域/Temporal/provider，B = controller/接口/契约**；共享文件（`app.module`/DI wiring、`package.json`、`prisma schema`）改动**互相打招呼**；冲突不可避免 → 到时适配（AI 适配即可）。走小 PR、功能分支。

**🔴 鉴权握手待 A 给**：JWKS 端点（公钥）+ token claim（`workspace_id`/`roles`）——B 去跟 A 敲定，否则生产鉴权不通。

**🔴 需早敲定的跨团队握手 —— 鉴权**：我们不做登录，靠 **A 平台签发 token + 我们 JWKS 验签**解出 workspace。要真跑通需 A 给两样：① JWKS 端点（公钥），② token claim 约定（`workspace_id` / `roles`）。**B 应早跟 A 敲定**，否则接口对接时鉴权通不了。

## 6. 需团队/用户拍板的开放决策

1. **QGO/机会/归因的数据主数据归本后端还是 SaaS 平台？**（v2 北极星 SAO→v3 QGO；触达执行已延后，触点数据来源未定）——决定出口层归属与排期。
2. **审批链 / Policy 引擎 / Kill Switch / RBAC / Data Rights 引擎宿主：本后端 vs 平台？**（PRD 归「Growth Core」但本后端不做身份系统）——最关键边界张力，阻塞合规硬门落地。
3. **展会 Algolia/RX 路线**（合规灰偏红、律师 DPIA 未过、换届即失效）：暂停加深等 DPIA，还是先做律师评估？
4. **付费→免费的经济反转**要不要正式写进 PRD v3 正本认账？（当前实现已免费优先，PRD 母本仍假设付费瀑布）
5. **前后端接口契约 + 职责 RACI**：尽快和「做 API 的人」对齐一版，避免两边各建一半或都不建。
