## 1. 产品负责人可以怎样提出需求

不需要先写 PRD、设计数据库或使用技术术语。最小输入是：

> 谁在什么场景遇到了什么问题，希望得到什么业务结果。

例如：

> 我希望客户上传公司资料后，能自动生成一个看起来专业的网站；生成失败时也不要丢掉已经填写的资料。

Codex 负责补齐场景、范围、失败恢复、事实边界和验收条件，再请产品负责人判断“这是不是我要的”。需求尚未确认前，主要模式是需求理解，不进入产品代码实施。

## 2. 需求合同

每个进入设计或实施的需求应形成以下合同。它可以写在 Issue、计划、PRD 或当前任务中，不强制新建常驻文档；承重事实仍写回其主题 Owner。

| 字段           | 要回答的问题                                    |
| -------------- | ----------------------------------------------- |
| 需求 ID        | 如何稳定引用和追踪这项需求                      |
| 目标用户       | 谁直接使用或受影响                              |
| 使用场景       | 在什么触发条件和上下文发生                      |
| 当前痛点       | 现在为什么无法完成目标                          |
| 期望业务结果   | 用户和业务得到什么变化                          |
| 主流程         | 用户从哪里开始，怎样完成                        |
| 失败与恢复     | 超时、取消、失败或部分成功后怎么办              |
| 权限与数据边界 | 谁能看、谁能做、数据怎样隔离                    |
| 事实与内容边界 | 哪些内容可使用，哪些不得推断或虚构              |
| 最小交付范围   | 本轮必须完成的最小闭环                          |
| 明确不做       | 哪些相邻能力不在本轮                            |
| 成功指标       | 怎样观察价值或风险                              |
| 验收条件       | 产品负责人可以重复执行的判断步骤                |
| 尚待决定       | 只有责任 Owner 能作出的选择                     |
| 追踪关系       | 对应 Scenario、Capability、Contract、测试和证据 |

### 2.1 可复制的需求输入

```markdown
## 我想解决的问题

- 谁遇到问题：
- 在什么场景：
- 现在的问题：
- 希望的结果：

## 我最在意

- 优先级：
- 不能接受的结果：
- 时间、成本或风险限制：
- 我会怎样判断满意：

## 已知材料

- 参考产品、页面或文件：
- 真实案例：
- 必须遵守的事实、品牌或合规要求：
```

不知道的字段可以留空。Codex 应通过仓库证据和最少量问题补齐，不能要求非技术产品负责人先完成技术分析。

## 3. 从自然语言到可验收合同的示例

对“上传资料后自动生成专业网站，失败不能丢资料”的需求，可转换为：

- 目标用户：首次建站的出海企业负责人。
- 业务结果：提交必要资料后看到基于真实输入生成的受控预览。
- 主流程：建立站点、保存资料、发起构建、观察状态、打开成功结果。
- 失败保护：构建失败不得删除站点、已保存资料或有效的上一结果。
- 恢复：显示可理解的失败类型和允许动作；满足条件时原地重试。
- 事实边界：未提供或无证据的资质、产能、客户案例和认证不得虚构。
- 最小范围：完成资料保存、构建、状态观察、成功预览、失败恢复的纵向闭环。
- 非目标：除非另行批准，不承诺公网发布、自定义域名、完整营销自动化或生产 SLA。
- 验收：按照指定点击路径可重复完成成功和失败恢复，并能识别哪些动作尚未接通。

这是需求编译示例，不表示上述能力当前已经实现。当前事实必须从[当前状态](../status/current.md)和真实合同核验。

## 4. 验收条件怎样写

好的验收条件描述可观察的用户结果、状态和证据，避免只描述内部实现。

| 不足的表达      | 更可验收的表达                                          |
| --------------- | ------------------------------------------------------- |
| “接口写好了”    | 有权限的目标用户能完成指定动作，并看到一致终态          |
| “测试通过”      | 成功、失败、取消、重试和旧结果保留路径均有可重复证据    |
| “支持权限”      | 无读权、无动作权、跨 Workspace 和会话过期都有明确结果   |
| “AI 生成得不错” | 指定真实/冻结样本满足事实、结构、质量和成本门，并可回滚 |
| “已经上线”      | 指明环境、版本、用户范围、发布时间、健康与回滚证据      |
| “不会丢数据”    | 明确哪些对象在何种失败下保留，并通过恢复场景验证        |

每项验收至少覆盖：

1. 用户从哪里进入；
2. 成功时执行什么、看到什么；
3. 失败、超时、取消或部分成功时看到什么；
4. 是否可以安全恢复或重试；
5. 权限和数据隔离是否正确；
6. 哪些动作仍不可用及原因；
7. 证据对应哪个提交、环境和时间。

涉及用户可见能力时，应从[场景目录](../governance/scenario-catalog.md)和[分析、测试与发布证据规范](../frontend/12-analytics-testing-and-release-evidence.md)选择相称的异常、权限、可访问性、安全和发布场景。

## 5. 开发准备门

一项需求进入实施前，至少满足：

- 目标用户、问题和业务结果可复述；
- 主流程、失败恢复和明确非目标已确认；
- 关键事实、权限、数据和合规边界明确；
- 所依赖的 API、对象或工作流有真实 Owner，未知处已标阻塞；
- 验收条件可以由非开发者观察，而不依赖阅读代码；
- 施工分支、worktree、写入边界和验证计划明确；
- 若有开放决定，已确认不会在施工中被执行者补猜。

不满足时，Codex 应保持在需求、研究或设计模式，并说明还缺哪个决定；不能用先写代码来替代需求确认。

## 6. 业务验收包

每次交付给产品负责人时，Codex 应提供：

1. 这次解决了什么用户问题。
2. 实际完成范围和明确未完成范围。
3. 用户从哪里进入、执行什么、看到什么。
4. 成功路径证据。
5. 失败、取消、超时和恢复路径证据。
6. 权限、数据、事实和安全边界。
7. 哪些按钮或能力真实接通，哪些仍被阻塞。
8. 已知限制和可能影响的用户。
9. 测试、审查、预览或运行证据的位置。
10. 若发布，怎样监控、停止和回滚。
11. 需要产品负责人作出的精确验收决定。

推荐的最终汇报骨架：

```markdown
## 结论

用户现在能完成什么；本轮没有承诺什么。

## 验收路径

入口 → 操作 → 成功结果 → 失败与恢复。

## 证据

提交/环境、测试、预览、合同和已知限制。

## 风险与回滚

影响范围、监控、停止条件和恢复方式。

## 请你决定

是否满足业务目标；若不满足，具体退回哪一项。
```

真实发布的 Release Bundle 有唯一规范，不在本指南复制。必须使用[分析、测试与发布证据规范](../frontend/12-analytics-testing-and-release-evidence.md)定义的 schema 和真实责任签发。

## 7. 产品负责人不需要看代码时应问什么

- 目标用户现在能否完成目标？
- 使用了真实数据和真实合同，还是 Mock、Demo 或目标设计？
- 失败后用户怎么办，旧数据和旧结果会不会丢？
- 谁能看、谁能执行，跨 Workspace 是否被阻止？
- 哪些能力还没接通，界面是否诚实表达？
- 证据来自哪个分支、提交、环境和日期？
- 合并是否完成，部署是否完成，两者是否被混淆？
- 发布后观察什么，异常时怎样停止和回滚？

这些问题比代码行数、测试数量或 AI 调用次数更接近业务验收。
