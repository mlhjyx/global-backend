这组指南把“只有业务需求、不会开发”转换成可审查的软件交付流程。产品负责人负责业务目标、优先级、风险取舍与验收；Codex 负责把自然语言需求逐步编译为范围、设计、技术方案、代码、测试和可核验结果。

它不建立第二套项目真值，也不授权安装 Obsidian、QMD、CodeGraph 或其他工具。当前边界、实现、决策和施工顺序仍以[项目文档门户](../README.md)列出的权威来源为准；仓库执行规则仍以 [AGENTS.md](../../AGENTS.md) 与 [CONTRIBUTING.md](../../CONTRIBUTING.md) 为准。

## 1. 按任务阅读

| 你现在要解决的问题                    | 阅读入口                                                 | 产出                             |
| ------------------------------------- | -------------------------------------------------------- | -------------------------------- |
| 不知道怎样把想法交给开发              | [非技术需求与业务验收](requirement-and-acceptance.md)    | 需求合同、验收路径、交付证据清单 |
| 不知道当前处于什么模式、谁负责什么    | [AI 辅助开发运行模型](operating-model.md)                | 模式、职责、升级规则、风险闭环   |
| 不知道 Obsidian、记忆或代码图能否相信 | [记忆、知识与派生认知政策](memory-and-context-policy.md) | 事实优先级、写入边界、工具试点门 |

第一次提出需求时，只需说明“谁遇到了什么问题、希望得到什么结果”。缺失内容由 Codex 通过需求编译过程补齐，并把需要产品负责人决定的少数问题单独列出。

## 2. 本指南负责与不负责

本指南负责：

- 给非技术产品负责人一个稳定、可重复的协作接口；
- 定义需求理解、研究、设计、实施、验收、发布等模式；
- 区分 Codex 应自行解决的问题与必须升级给产品负责人的问题；
- 规定知识、记忆、语义检索和代码图不得覆盖权威事实；
- 把常见项目风险转成开发前置门和验证动作。

本指南不负责：

- 改写产品边界、当前状态、架构、ADR 或路线；
- 替代 API、Schema、代码、CI、运行时或 Release Bundle；
- 证明某项能力已经实现、部署或可供真实用户使用；
- 代替产品、设计、隐私、安全、运营或发布责任人的签字；
- 授权安装、连接或写入任何外部知识与记忆工具。

## 3. 事实冲突时怎么处理

先按[术语、事实状态与责任词典](../governance/terminology-and-status.md)确定主题 Owner，不按文件更新时间、篇幅或 AI 回答的自信程度判断。指南、聊天记录、Obsidian 笔记、搜索结果和代码图只能帮助定位证据；发生冲突时，回到对应权威文件、机器合同、当前分支代码和运行证据。

无法判定时，Codex 必须将其标为未知或冲突，说明缺失证据，并请求相应 Owner 决定，不能补猜。
